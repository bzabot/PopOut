## Build Order (step by step)

Work through these in order. Each step builds on the previous ones.

### Part A: The Game Engine

#### Step 1 — Move

Define what a move is. In PopOut, a player can either **drop** a disc into a column from the top, or **pop** one of their own discs from the bottom.

A move is just two things: the type (drop or pop) and the column (0-6). Make it simple and immutable — once created, it shouldn't change.

> **Hint:** A frozen dataclass with an enum for the move type works well. Add validation so nobody can create a move with column 9.

#### Step 2 — Board

The board is a 6x7 grid. You need to represent the state and provide operations to change it:

- **Drop:** Place a disc at the top of a column — gravity pulls it to the lowest empty row.
- **Pop:** Remove a disc from the bottom of a column — everything above slides down by one.

Key decisions:
- Row 0 = top, row 5 = bottom (or the other way — just be consistent everywhere).
- The board should be **immutable**: `drop` and `pop` return a *new* board, leaving the original unchanged. This is critical for MCTS, which needs to explore many game states without corrupting the current one.
- Two boards with the same grid should be equal and hashable (needed for the three-repetition rule).

> **Hint:** Store the grid as a numpy array. Copy it on construction so the caller can't mutate your internal state.

#### Step 3 — Rules

This is the most complex part of the game engine. You need:

- **Legal moves:** Which moves can a player make right now? Drops into any non-full column, pops from any column where the bottom disc is yours.
- **Win detection:** Check all possible lines of 4 (horizontal, vertical, both diagonals). On a 6x7 board there are 69 possible windows.
- **The three PopOut-specific rules:**
  1. **Simultaneous win on pop:** If popping creates four-in-a-row for *both* players, the player who popped wins.
  2. **Full board:** When the board is full, the current player chooses to pop or declare a draw. (Handle this in the game loop, not here.)
  3. **Three-repetition draw:** If the same board state appears 3 times, the game is a draw.

> **Hint:** For win detection, iterate over every possible starting position and check the 4 directions (right, down, down-right, down-left). Only check directions that fit within the board.

#### Step 4 — Game Loop

Wire everything together so you can actually play:

- Support three modes: human vs human, human vs AI, AI vs AI.
- Use a **pluggable player** design: the game loop doesn't care *how* a player picks a move — it just calls a function that takes the board and legal moves and returns a move.
- Print the board after each move (ASCII: `X` for player 1, `O` for player 2, `-` for empty).
- Track the state history for the three-repetition rule.
- Handle the full-board draw offer: when the board is full, ask the current player if they want to pop or draw.

> **Hint:** Define a player as any function with the signature `(board, legal_moves) -> move_or_None`. A human player prompts for input. An AI player calls MCTS. This way the game loop stays clean.

### Part B: Monte Carlo Tree Search

#### Step 5 — UCT Formula

The UCT score balances **exploitation** (play moves that win often) with **exploration** (try moves we haven't tested much):

```
UCT = (wins / visits) + C * sqrt(ln(parent_visits) / visits)
```

Where `C` is the exploration constant (typically sqrt(2)). This is a small, self-contained function.

#### Step 6 — MCTS Node

A node in the search tree tracks:
- How many times it's been visited
- How many wins from this position
- Which move led to this node
- Its parent and children

Keep it simple — just a class with these attributes plus a few helpers like `is_leaf()` and `best_child_by_visits()`.

#### Step 7 — Standard MCTS

The core algorithm has four phases that repeat for N iterations:

1. **Select:** Start at the root and walk down the tree, picking the child with the best UCT score at each level, until you reach a leaf.
2. **Expand:** Add all legal moves as children of that leaf.
3. **Simulate:** From the new position, play random moves until the game ends (win, loss, or draw).
4. **Backpropagate:** Walk back up to the root, updating visit counts and win counts.

After all iterations, return the root's child with the **most visits** (not the highest win rate — visits are more stable).

> **Hint:** The simulation (random playout) doesn't need to be smart — just pick random legal moves until the game ends. The intelligence comes from doing many simulations.

#### Step 8 — MCTS Variants

The assignment asks you to explore different strategies, not just the standard MCTS. Implement at least two variants, for example:

- **Top-k selection:** Instead of always picking the single best UCT child, randomly pick among the top-k. This adds diversity to the search.
- **Progressive widening:** Don't expand all children at once — only add a new child when the node has been visited enough times. This focuses the search early on.

You'll compare these variants in the notebook to see which plays better.

### Part C: Decision Trees (ID3)

#### Step 9 — Discretizer

ID3 works on **categorical** data, but the iris dataset has continuous numbers. You need to convert them into bins.

Implement at least one binning strategy:
- **Equal-width:** Divide the range [min, max] into N equal-sized bins.
- **Equal-frequency:** Each bin gets roughly the same number of data points.

> **Hint:** `fit()` learns the bin edges from training data, `transform()` maps values to bin indices. The iris dataset has 4 features — discretize each one independently.

#### Step 10 — Entropy and Information Gain

These are the metrics that drive ID3's split decisions:

- **Entropy:** Measures how "mixed" a set of labels is. Formula: `-sum(p * log2(p))` for each class proportion `p`. Pure set = 0, perfectly mixed = 1 (for two classes).
- **Information gain:** How much entropy decreases after splitting on an attribute. `IG = entropy(parent) - weighted_avg(entropy(children))`.

ID3 picks the attribute with the highest information gain at each step.

#### Step 11 — ID3 Algorithm + Tree Node

The ID3 algorithm builds a decision tree recursively:

1. If all labels are the same → create a leaf with that label.
2. If no attributes left → create a leaf with the majority label.
3. Otherwise → find the attribute with the highest information gain, split the data on it, and recurse for each subset.

The tree node is simple: either it's a **leaf** (has a label) or an **internal node** (has an attribute to split on and children for each possible value).

Also implement:
- **Prediction:** Walk down the tree following the feature values until you hit a leaf.
- **Cross-validation:** Split data into N folds, train on N-1, test on 1, repeat. Report accuracy per fold.

> **Hint:** Add optional `max_depth` and `min_samples_split` parameters for regularization — they'll help when the PopOut tree gets too deep.

#### Step 12 — Tree Visualizer

You need to show your trees in the notebook. Build two renderers:

- **ASCII:** Indented text output, good for quick inspection.
- **Matplotlib:** A proper tree diagram with boxes and lines, good for the slides.

> **Hint:** For the PopOut tree (which can be huge), add a `max_display_depth` parameter so you only render the first few levels.

## Common Pitfalls

- **Mutable board state:** If your board isn't truly immutable, MCTS will corrupt game states during search. Test this early.
- **Wrong win detection:** Missing a diagonal direction or off-by-one on the grid boundaries. Check all 69 windows.
- **Pop gravity:** After popping, *all* discs above must shift down, not just the one directly above.
- **Forgetting the simultaneous win rule:** A pop can create four-in-a-row for both players. The popper wins.
- **ID3 on continuous data:** If you forget to discretize the iris features, ID3 won't work (it needs categorical splits).
- **Giant PopOut tree:** Without `max_depth`, the PopOut decision tree can be enormous. Use cross-validation to find the sweet spot.
